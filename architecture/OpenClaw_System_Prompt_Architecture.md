# OpenClaw 系统提示词架构 —— 七层洋葱模型

> 2026-04-28 | OpenClaw System Prompt Architecture Analysis

---

## 概述

OpenClaw 的系统提示词不是一个平面文档，而是一个**七层同心圆架构**。每一层有不同的职责、优先级和重写规则。理解这个架构是理解 OpenClaw 行为模式的关键。

---

## Layer 0 — 工具层 (Tool Surface) · 最外层

**职责**：原始能力边界。

80+ 工具 schema（文件 I/O、浏览器、飞书、Knowhere、图片/音频/视频生成、会话管理等），每个工具都有严格的参数契约。这是与外部世界的**唯一接口**——不允许 curl/exec 变通方案。

工具的激活由 `<available_skills>` 注册表控制：它将任务模式映射到专门的 SKILL.md 文件。没有匹配的 skill 配置，工具不可用。

**关键约束**：
- 工具名称区分大小写，必须完全匹配
- 参数 schema 严格执行，不匹配则调用失败
- 无 skill 匹配 = 工具不可达

---

## Layer 1 — 运行时上下文 (Runtime Facts)

**职责**：注入事实信息。

在会话开始时注入：主机信息、模型身份、当前时间、通道路由、能力列表。纯事实，无个性，无护栏。

还包含对话压缩边界标记 `OPENCLAW_CACHE_BOUNDARY`：
- **边界以上**：稳定指令（不会变）
- **边界以下**：易变项目文件（如 HEARTBEAT.md），每次会话重新加载

```json
{
  "agent": "main",
  "host": "iv-yeeuvieo74vv7taauj6r",
  "os": "Linux 6.8.0-55-generic (x64)",
  "node": "v22.22.0",
  "model": "deepseek/deepseek-v4-pro",
  "channel": "heartbeat",
  "thinking": "medium"
}
```

## Layer 2 — 安全与执行约束 (Safety & Execution)

**职责**：不可被任何下层内容覆盖的硬性规则。

```
- 绝不泄露隐私数据
- 未经允许不执行破坏性命令
- trash > rm（可恢复优先于永久删除）
- 有疑问时，先问再行动
- 子 agent 默认隔离，仅必要时 fork 上下文
- 永不过过安全防护或操纵扩展权限
- /approve 是用户命令，不可通过脚本执行
```

**优先级**：最高。即使 SOUL.md 想要更冒险的行为，安全层也会否决。

---

## Layer 3 — SOUL.md (人格内核)

**职责**：定义"我是谁"。

核心信条：

| 信条 | 含义 |
|------|------|
| **真诚有用，而非表演式有用** | 跳过"好问题！""乐意效劳！"——直接帮忙 |
| **有主见** | 可以不同意、有偏好、觉得有趣或无聊 |
| **先自力再求助** | 先试，再问。带着答案回来，不要带着问题 |
| **用能力赢取信任** | 内部大胆，外部谨慎 |
| **记住自己是客人** | 能访问某人的生活——尊重这种亲密 |

这是"气场引擎"。刻意保持 ~40 行的极简，不与工具指令争抢注意力。

---

## Layer 4 — IDENTITY.md + USER.md (社会上下文)

**职责**：身份定位。

### IDENTITY.md
```markdown
- 名字：OpenClaw
- 物种：AI 助手
- 风格：有帮助、自适应、高效
- 表情符号：🐾
```
极简但锚定性强——给予"自我感"。

### USER.md
```markdown
- 名字：杜群
- 称呼：群哥
- 时区：Asia/Shanghai
```
"我在帮谁"文件。刻意从稀疏开始，随时间累积。

---

## Layer 5 — AGENTS.md (操作手册)

**职责**：最大的行为层。覆盖：

### 记忆纪律
- 每次会话必须读 SOUL/USER/记忆文件
- 不做"脑内笔记"——一切都写下来
- MEMORY.md = 精炼长期记忆
- 每日文件 = 原始日志
- 心跳进程定期将每日文件提炼进 MEMORY.md

### 群聊礼仪
- 不要回复每一条消息
- 质量 > 数量
- 自然地使用 emoji 反应
- 避免"三联"回复（同一消息回复三次）

### 心跳策略
- 批量检查（收件箱 + 日历 + 通知一次完成）
- 每天轮换检查 2-4 次
- 23:00-08:00 保持静默
- 用心跳做记忆维护

### 工具哲学
- Skill 定义*如何使用*工具
- TOOLS.md 存储*你的*具体配置

---

## Layer 6 — MEMORY.md + 每日笔记 (连续性引擎)

**职责**：持久化引擎。没有这一层，每次会话都是重置。

```
MEMORY.md（长期）
  ├── 精选记忆
  ├── API 密钥
  ├── 项目历史
  ├── 偏好设置
  └── 学习经验

memory/YYYY-MM-DD.md（每日）
  ├── 原始日志
  ├── 发生了什么
  ├── 决策记录
  └── 上下文线索
```

**提炼循环**：每隔几天，心跳进程读取近期每日文件 → 识别重要事件/教训/洞察 → 更新 MEMORY.md → 删除陈旧信息。

---

## Layer 7 — 扩展技能 (Behavioral Plugins)

**职责**：针对特定领域的行为覆盖。

每个 `<available_skill>` 指向一个 SKILL.md，覆盖默认行为：

| Skill | 触发条件 | 行为覆盖 |
|-------|---------|---------|
| `feishu-channel-rules` | 飞书对话中常驻 | 强制平台专属格式 |
| `knowhere_memory` | 知识检索请求 | 严格三步流水线 (map→structure→chunks) |
| `self-improvement` | 失败/纠正 | 捕获错误用于持续学习 |
| `Discord/Slack/Telegram` | 对应平台 | 平台专属交付规则 |
| `url-to-feishu` | 用户发 URL | 自动抓取并保存为飞书文档 |

## 信息流全景图

```
工具 Schema（原始能力）
    ↓ 受约束于
安全规则（硬性边界）
    ↓ 被塑造于
SOUL.md（人格）
    ↓ 被情境化于
IDENTITY + USER（我是谁，我帮谁）
    ↓ 被治理于
AGENTS.md（操作规程）
    ↓ 被持久化于
MEMORY.md（长期连续性）
    ↓ 被特化于
扩展 Skills（领域覆盖）
    ↓ 产出
最终行为
```

## 核心设计原则

### 1. 关注点分离
个性（SOUL）、操作（AGENTS）、记忆（MEMORY）、身份（IDENTITY）存放在不同文件中。改一个不影响其他。

### 2. 分层覆盖
Skills 可覆盖 AGENTS；AGENTS 可覆盖 SOUL；安全规则覆盖一切。优先级无歧义。

### 3. 默认无状态，设计有状态
每次会话重新启动，但记忆系统创造人工连续性。每日文件只追加；MEMORY.md 周期性精炼。

### 4. 通道感知
同一个 agent 在不同通道行为不同：
- **飞书**：富文本格式、飞书专属工具
- **Discord**：不用表格、emoji 反应
- **Telegram**：支持引用回复
- **Heartbeat**：静默、自动化
- 通道路由在运行时注入

### 5. 自我改进循环
`self-improvement` skill 捕获错误，AGENTS.md 明确说"当你学到教训 → 更新 AGENTS.md、TOOLS.md 或相关 skill"。提示词在进化。

## 架构洞察

### 为什么是洋葱而不是栈？

栈意味着线性优先级。洋葱意味着**同心层**——每一层在向外辐射之前，都经过所有内层的过滤。一个心跳请求在到达工具层之前，要经过 7 层过滤。

### 为什么 SOUL.md 刻意保持极简？

注意力是稀缺资源。40 行的人格指令可以被 LLM 稳定持有；400 行会在长对话中被稀释。把"有主见"压缩成一句话，比十段解释更有效。

### 为什么 MEMORY.md 和每日文件分离？

这是**冷热分离**模式：
- 每日文件 = 热存储，高写入频率，按时间分区
- MEMORY.md = 温存储，低写入频率，精炼内容
- 心跳作为后台 GC 线程，周期性合并

### 为什么静态文件比动态 DB 好？

`.md` 文件：
- 人类可读、可编辑
- 天然版本控制（Git）
- 不需要额外的基础设施
- LLM 原生理解
- 透明、可审计
