## 📌 一句话总结

这篇论文通过逆向工程 Claude Code（v2.1.88）的 TypeScript 源码，系统性地拆解了其架构设计空间，并与 **OpenClaw** 进行对比，揭示了相同设计问题在不同部署上下文下的不同答案。

---

## 🏗️ 核心架构：一个 while 循环 + 五层外围系统

论文的核心发现：Claude Code 的核心惊人地简单——一个 `while` 循环调用模型、运行工具、重复。但 **98.4% 的代码** 都在这个循环外围的子系统中。

```
┌─────────────────────────────────────────────────────────────┐
│                        七层安全防御                           │
│  (工具预过滤 → 拒绝优先规则 → 模式约束 → ML分类器 → 沙箱 →   │
│   恢复不恢复权限 → 钩子拦截)                                  │
├─────────────────────────────────────────────────────────────┤
│                        智能体循环                             │
│              queryLoop(): 调用模型 → 运行工具 → 重复          │
├─────────────────────────────────────────────────────────────┤
│                    五层上下文压缩管道                          │
│  (预算缩减 → 剪切 → 微压缩 → 上下文折叠 → 自动压缩)           │
├─────────────────────────────────────────────────────────────┤
│                    四种扩展机制                                │
│         MCP服务器 / 插件 / 技能 / 钩子                        │
├─────────────────────────────────────────────────────────────┤
│                    子智能体隔离边界                            │
│         AgentTool → 独立queryLoop() → 摘要返回                │
├─────────────────────────────────────────────────────────────┤
│                    仅追加持久化存储                            │
│              JSONL 会话记录 + CLAUDE.md 记忆                  │
└─────────────────────────────────────────────────────────────┘
```

## 🎯 五大设计价值

| 价值 | 核心含义 | 架构体现 |
|:---|:---|:---|
| **人类决策权威** | 人类保留最终决策权 | 拒绝优先默认、渐进信任谱系、权限可审计 |
| **安全与隐私** | 系统保护用户即使用户犯错 | 七层独立防御、自动模式 ML 分类器、沙箱 |
| **可靠执行** | 做用户真正意图的事 | 五层上下文压缩、优雅恢复、生成与评估分离 |
| **能力放大** | 实质增加人类产出 | 最小脚手架（仅 1.6% 决策代码）、最大操作工具 |
| **情境适应性** | 随时间适应用户 | 透明文件配置、渐进信任、用户可见的记忆 |

> **关键洞察**: Anthropic 发现用户批准 93% 的权限提示后，**没有增加更多警告**，而是重构问题——定义边界让智能体在边界内自由工作。

## 📜 十三项设计原则（精华）

| # | 原则 | 一句话解释 |
|:---|:---|:---|
| 1 | **拒绝优先，人工升级** | 不认识的动作 → 问人，不是允许也不是阻止 |
| 2 | **渐进信任谱系** | 不是固定权限，是用户随时间穿越的光谱 |
| 3 | **纵深防御** | 不是单一安全边界，是七层重叠技术 |
| 4 | **外部化可编程策略** | 策略不是硬编码，是带生命周期钩子的配置 |
| 5 | **上下文作为稀缺资源** | 不是单次截断，是五层渐进压缩管道 |
| 6 | **仅追加持久状态** | 不修改历史，只追加日志 |
| 7 | **最小脚手架** | 让模型自由推理，投资确定性基础设施 |
| 8 | **价值高于规则** | 不是刚性程序，是确定性护栏支持的判断 |
| 9 | **可组合多机制扩展** | 不是单一 API，是四种不同成本的分层机制 |
| 10 | **可逆性加权风险评估** | 只读/可逆动作更轻监督 |
| 11 | **透明基于文件的配置** | 不是不透明数据库，是用户可见的可版本控制文件 |
| 12 | **隔离子智能体边界** | 子智能体隔离运行，摘要返回 |
| 13 | **优雅恢复和弹性** | 错误时静默恢复，只为不可恢复情况保留人类注意力 |

## 🔐 权限系统：七种模式 + ML 分类器

### 七种权限模式（渐进信任光谱）

```
plan → default → acceptEdits → auto → dontAsk → bypassPermissions → bubble
 ↑                                              ↑
最高人工控制                                    最高自动化
（先计划再执行）                                （子智能体内部专用）
```

| 模式 | 核心行为 |
|:---|:---|
| **plan** | 模型必须先创建计划，用户批准后才能执行 |
| **default** | 大多数操作需要用户批准 |
| **acceptEdits** | 工作目录内编辑自动批准，shell 命令仍需批准 |
| **auto** | ML 分类器评估边界案例 |
| **dontAsk** | 不提示用户，但拒绝规则仍执行 |
| **bypassPermissions** | 跳过大多数权限提示，安全关键检查仍保留 |
| **bubble** | 子智能体权限升级到父终端（内部专用） |

### 自动模式 ML 分类器（两阶段）

```
快速过滤（规则启发式）→ 明显安全/不安全的请求快速通过
           ↓
思维链评估（完整 LLM 推理）→ 边界案例深度推理
```

**关键设计**: 拒绝不是硬停止，而是**路由信号**——模型接收拒绝原因，修订方法，在下一循环迭代尝试更安全的替代方案。

## 🔧 四种扩展机制对比

| 机制 | 上下文成本 | 注入点 | 独特能力 |
|:---|:---|:---|:---|
| **MCP 服务器** | 高（工具模式） | model() 工具池 | 外部服务集成，多传输协议 |
| **插件** | 中等（可变） | 全部三个点 | 多组件打包 + 分发 |
| **技能** | 低（仅描述） | assemble() 上下文注入 | 领域特定指令 + 元工具调用 |
| **钩子** | 默认零成本 | execute() 执行拦截 | 生命周期拦截 + 事件驱动自动化 |

**27 种钩子事件类型**，覆盖：工具授权、会话生命周期、用户交互、子智能体协调、上下文管理、工作区事件、通知。

## 🤖 子智能体：隔离边界设计

```
父智能体上下文窗口 ← 摘要文本（仅）
         ↓
    [AgentTool 调用]
         ↓
    子智能体 queryLoop() —— 独立上下文窗口
         ↓
    完整对话历史 → 侧链文件（单独存储）
```

**设计原理**:
- 隔离上下文窗口 → 防止子智能体膨胀父上下文
- 摘要仅返回 → 上下文资源管理
- 独立权限边界 → 通过 `bubble` 模式升级
- 侧链存储 → 每个子智能体单独文件

## ⚖️ Claude Code vs OpenClaw：六大设计维度对比

| 设计维度 | Claude Code（商业 CLI 工具） | OpenClaw（开源网关） |
|:---|:---|:---|
| **安全评估位置** | **每次动作评估**：七层防御，细粒度 | **边界级访问控制**：网关层面控制 |
| **执行引擎** | **单一 CLI 循环**：`queryLoop()` 统一所有入口 | **嵌入式运行时**：网关控制平面内嵌 |
| **上下文扩展** | **上下文窗口压缩**：五层管道最大化单会话 | **网关范围能力注册**：跨服务资源协调 |
| **信任边界** | 用户-工具直接信任，渐进自动化 | 服务-服务委托信任，OAuth/令牌基础 |
| **扩展性模型** | 四种机制，上下文成本分层 | 网关插件架构，统一服务接口 |
| **部署形态** | 本地 CLI，直接文件系统访问 | 网络网关，多通道接入 |

### 共同收敛点

两者都需要解决：工具授权、上下文管理、扩展机制、多智能体/服务协调。

### 差异根源

| 因素 | Claude Code | OpenClaw |
|:---|:---|:---|
| 部署环境 | 开发者本地环境 | 服务器/云环境 |
| 用户假设 | 技术用户 | 广泛用户 |
| 安全模型 | 防止 AI 误操作 | 防止服务滥用 |
| 产品目标 | 编码效率 | 通用助理 |
| 信任动态 | 用户-AI 渐进信任 | 服务提供商委托信任 |

## 🔮 未来六个开放设计方向

### 方向 1：可观测性-评估差距
**问题**: 能观察行为，但缺乏系统性评估是否达成目标的机制  
**开放问题**: 如何构建运行时评估基础设施？如何将环境地面实况从诊断提升为评估信号？

### 方向 2：跨会话持久性
**问题**: 跨会话的长期记忆、学习和对齐机制有限  
**开放问题**: 如何在保护隐私的同时实现跨会话学习？会话恢复不恢复权限的安全-体验张力如何解决？

### 方向 3：Harness 边界演进
**问题**: 随着模型能力增强，推理与执行的边界应如何移动？  
**开放问题**: "最小脚手架"原则是否持续最优？何时将更多逻辑迁移到脚手架侧？

### 方向 4：时间范围扩展
**问题**: 优化短期任务，长期连贯性不足  
**开放问题**: 如何支持数周/数月时间范围？如何避免"短期修复、长期技术债务"？

### 方向 5：治理机制
**问题**: 多智能体系统和自动模式下决策权威分配缺乏正式框架  
**开放问题**: ML 分类器决策如何审计？权限升级的正式安全分析？

### 方向 6：长期能力保存（最发人深省）
**问题**: 架构未将长期人类能力保存作为共同价值  
**研究发现**:
- Anthropic 自身研究："监督悖论"——过度依赖 AI 风险萎缩监督技能
- 独立研究：AI 辅助开发者理解测试得分**低 17%**

**开放问题**: 如何在架构中显式设计支持人类学习和理解？短期能力放大 vs 长期人类发展的 trade-off 如何量化？

> **关键反思**: Claude Code 约 27% 的任务是"无工具则不会尝试"的工作——这是能力放大的胜利，但也可能是长期依赖的风险。

---

## 💡 对 OpenClaw 的启示

1. **上下文管理**: Claude Code 的五层压缩管道值得借鉴，OpenClaw 作为网关可能需要更复杂的跨服务上下文协调
2. **安全模型**: OpenClaw 的边界级控制 vs Claude Code 的逐动作评估，两者互补而非替代
3. **扩展机制**: Claude Code 的"四种机制分层"设计可为 OpenClaw 的插件系统提供参考
4. **长期能力保存**: 这是两者都未充分解决的问题，可能是未来关键差异化方向
5. **渐进信任**: OpenClaw 作为多用户网关，如何在服务级别实现类似的用户级渐进信任？
